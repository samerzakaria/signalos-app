# SignalOS.NET Concept Parity Audit

Date: 2026-06-29

Scope: `signalos-app` compared against the adjacent `SignalOS.NET` repository as a concept and behavior source. This is not a request to copy ABP or force .NET output. The target is SignalOS behavior parity: modules, hooks, agents, gates, stages, proof, refusal, audit, and closeout remain SignalOS concepts while generated products can use any provable technology.

## Parity Rule

SignalOS product delivery must be technology-independent:

- User technology choices are capabilities, not framework locks.
- ABP and .NET are allowed only when explicitly selected or when an adapter can prove they are the right product technology.
- Normal infrastructure such as PostgreSQL, SQL Server, MySQL, SQLite, Redis, and Memcached is allowed when requested or selected by an adapter.
- If a requested technology cannot be proved by a native adapter or agent output, delivery must stay partial with a clear blocker instead of silently switching stacks.
- Common-adopter adapter choices are grounded in current public usage signals, including the 2025 Stack Overflow Developer Survey technology section: https://survey.stackoverflow.co/2025/technology/

## SignalOS.NET Concepts Observed

The source repo contains a broader operating model than a product generator:

- modules for constitutions, beliefs, waves, trust tiers, hooks, observability, worktree sync, audit trail, security, auth server, gateway, worker, and web;
- command surface for agent run, attach, audit, brain, build, bundle, careful, catalog, checkpoint, cost, debrief, defer, deliver, deploy, devex, doctor, feature gate, gate, grade, guard, handoff, hooks, investigate, observe, onboard, pack, plan, preamble, pre-wave, product, publish, recall, recover, release readiness, review, session, ship, test, trace, upgrade, verify, and wave review;
- validator behaviors for governance obligations, bypass detection, product proof, and release readiness;
- product factory adapters including ABP Angular, minimal API, React/Vite, and existing repo;
- evidence-first product proof scenarios and closeout rather than generation-only success.
